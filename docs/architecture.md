# 服务化重构方案

## 1. 背景

当前项目由两部分组成：

- Python 脚本负责扫描本地目录中的 Git 仓库并输出 CSV。
- React 看板负责读取本地 `data` 目录或手工上传的 CSV 进行展示。

这套模式适合离线分析，但不支持以下能力：

- 用户输入一个 Git 地址后自动下载代码并分析。
- 记录分析任务状态和历史结果。
- 管理项目、分支、本地缓存和删除策略。
- 页面刷新或服务重启后直接恢复分析结果。

## 2. 重构目标

将项目升级为单机服务，支持：

- 新增项目时输入 Git 地址。
- 为项目配置一个或多个分支。
- 后端自动 clone 或 fetch 仓库并执行分析。
- 保存分析结果和任务状态。
- 前端直接展示“最新结果”或历史分析结果。

## 3. 目标架构

```text
Frontend (React)
  |
  v
API Service (FastAPI)
  |- Project Manager
  |- Branch Manager
  |- Analysis Job Runner
  |- Result Query
  |
  v
Analyzer (Python)
  |- Git log parser
  |- Metrics aggregator
  |- JSON/CSV exporter
  |
  v
Storage
  |- SQLite metadata
  |- local repo cache
  |- JSON result files
```

## 4. 模块职责

### 4.1 Frontend

负责：

- 创建项目。
- 管理项目分支。
- 发起更新或重新分析。
- 查看分析状态。
- 展示最新结果和历史结果。

不负责：

- 直接访问 Git。
- 直接读取本地仓库目录。
- 直接拼装 AI 请求密钥之外的仓库分析流程。

### 4.2 API Service

负责：

- 校验 Git 地址和分支参数。
- 维护项目、分支和任务元数据。
- 管理本地仓库缓存目录。
- 调用分析器并写入结果文件。
- 提供查询接口给前端。

### 4.3 Analyzer

负责：

- 读取本地仓库或远端分支引用的 Git 历史。
- 计算月份、作者、模块、工程维度统计。
- 输出统一 JSON 结果。
- 按需导出 CSV。

分析器不再直接负责：

- 扫描整个源码目录。
- 面向终端的交互式输出。
- 作为系统唯一入口。

## 5. 存储策略

采用“仓库缓存”和“分析结果”分离的策略。

### 5.1 本地仓库缓存

用途：

- 避免每次分析都重新 clone。
- 支持多个分支复用同一份仓库副本。

策略：

- 一个项目只保留一个本地仓库目录。
- 更新项目时先 `git fetch`，再针对目标分支分析。

### 5.2 分析结果

用途：

- 页面刷新后直接展示。
- 服务重启后恢复历史分析状态。
- 支持历史运行记录查询。

策略：

- 主存储格式使用 JSON。
- 需要兼容导出或旧逻辑时再生成 CSV。
- 每次分析生成一份独立结果文件，不覆盖历史记录。

### 5.3 元数据

采用 SQLite 保存：

- 项目信息。
- 分支信息。
- 分析任务状态。
- 最近一次成功分析的索引。
- 本地仓库路径和最近 commit。

## 6. 项目与分支管理原则

本系统采用以下建模方式：

- 项目 = 一个 Git 仓库。
- 分支 = 项目下的一个分析目标。
- 运行记录 = 某个分支在某个 commit 上的一次分析结果。

不将“每个分支”单独建模成一个项目，原因如下：

- 同一个仓库的分支天然属于同一个业务对象。
- 本地仓库缓存可以复用，避免重复 clone。
- 删除项目时可以统一清理所有分支和结果。

## 7. 任务生命周期

一次分析任务的标准流程如下：

1. 前端提交项目分支分析请求。
2. 后端创建任务记录，状态为 `queued`。
3. 任务执行器检查本地仓库是否存在。
4. 不存在则 `clone`，存在则 `fetch`。
5. 解析目标分支对应的最新 commit SHA。
6. 若该 commit 已有成功结果，可直接复用结果或按参数决定重新分析。
7. 若没有结果，则执行分析器生成 JSON 结果。
8. 写入结果文件，更新任务状态为 `succeeded`。
9. 前端轮询任务状态并读取结果。

失败状态至少应包含：

- `clone_failed`
- `fetch_failed`
- `analyze_failed`
- `canceled`

## 8. 删除与清理策略

删除动作分为 3 类，避免一个“删除”按钮做过多事情：

### 8.1 删除分析记录

- 删除某次或某批历史结果文件。
- 保留项目配置和本地仓库缓存。

### 8.2 清理本地代码缓存

- 删除本地仓库目录。
- 保留项目和历史结果。
- 下次更新时重新 clone。

### 8.3 彻底删除项目

- 删除项目配置。
- 删除该项目下所有分支记录。
- 删除所有分析结果。
- 删除本地仓库缓存。

## 9. 初始目录结构建议

```text
docs/
src/
backend/
  app/
    api/
    models/
    services/
    analyzer/
    storage/
storage/
  repos/
    <project_id>/
      repo/
  results/
    <project_id>/
      <branch_name>/
        <run_id>.json
        <run_id>.csv
  app.db
```

其中：

- `backend/app/analyzer/` 用于承接从 `analyze_repos.py` 抽出的核心分析逻辑。
- `storage/` 为运行时数据目录，不建议直接放进前端源码目录。

## 10. 结果缓存原则

页面刷新时：

- 直接读取该分支最近一次成功的结果。
- 不自动触发重新分析。

点击“更新项目”时：

- 执行 `git fetch`。
- 检查最新 commit 是否变化。
- 如无变化，返回“已是最新结果”。
- 如有变化，创建新的分析运行记录。

## 11. 实施阶段

### Phase 1: 后端基础能力

- 抽离分析器为可复用 Python 模块。
- 建立 SQLite 表结构。
- 提供项目、分支、任务、结果查询接口。
- 支持单机单任务串行执行。

### Phase 2: 前端接入

- 增加项目管理页。
- 增加分支管理和任务状态展示。
- 改造现有看板，使其消费服务返回的 JSON 结果。

### Phase 3: 进阶能力

- 私有仓库认证。
- 定时更新。
- 历史结果对比。
- 结果导出和清理策略。
- AI 分析迁移到后端代理层。

## 12. MVP 范围

第一版建议只支持：

- HTTPS 公有仓库。
- 手动新增项目。
- 手动触发更新。
- 分支级分析。
- 单机部署。
- JSON 结果展示。

暂不优先处理：

- 分布式任务队列。
- 大规模并发分析。
- 多用户权限模型。
- 每次运行的全量审计功能。
